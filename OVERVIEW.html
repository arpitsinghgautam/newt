Part 1A primer: from CPUs to Triton

1.1 Why GPUs exist

A CPU (central processing unit) has a few smart cores optimized to run one instruction stream very fast. A GPU (graphics processing unit) has thousands of simple cores, grouped into a few dozen SMs (streaming multiprocessors). It wins whenever you do the same simple operation on millions of numbers at once, which is exactly what neural networks do.

1.2 Kernels, threads, warps, blocks

A kernel is a function launched on the GPU across many parallel workers. You write the body once; a grid of workers runs it. One worker is a thread; 32 threads execute in lockstep as a warp; up to ~1024 threads form a thread block, which runs on one SM and can share fast memory and synchronize.

1.3 The memory pyramid

Two consequences drive everything:

  • Coalescing: when a warp's 32 threads read 32 adjacent numbers, the hardware serves them in one wide transaction. Scattered reads can cost 32.
  • Latency hiding: the GPU keeps many more threads resident than executing, so while some wait on memory, others compute. A kernel that cannot keep enough work in flight leaves the arithmetic units idle.

1.4 Why ML needs custom kernels: fusion

1.5 CUDA, Triton, Helion

CUDA (NVIDIA's GPU programming platform) gives full control in a C++ dialect, and demands full control back: manual thread indexing, shared-memory management, synchronization barriers, and tensor cores (dedicated matrix-multiply hardware, ~16x faster than ordinary arithmetic for that job) driven through special instructions. A production CUDA matmul is hundreds of lines, and one misplaced barrier means silently wrong answers.

Triton (an open-source compiler, originally from OpenAI) raised the level: you program a block of data, not threads. You write "load this block, subtract its max, exponentiate, divide by its sum, store it" and the compiler decides how elements map to threads, how the reduction communicates, where barriers go, and how to reach the tensor cores. Internally it is a large industrial compiler: Python is lowered through an IR (intermediate representation) and the MLIR and LLVM frameworks down to PTX (NVIDIA's portable GPU assembly), then to SASS (the real machine code).

Helion (from PyTorch) sits one level higher still: PyTorch-like tile code goes in, and the compiler generates the Triton kernel and autotunes it, searching block sizes and other knobs automatically, verifying each candidate, and caching the fastest.

Everything here compiles JIT (just-in-time): a kernel is compiled on first call for each combination of data types and compile-time constants, then cached.

Part 2The problem statement

Triton and Helion are hundreds of thousands of lines. You cannot read them. The best way to own the ideas inside a big system is to rebuild it small, the way nano-vllm rebuilt the vLLM inference engine in ~1,200 lines with comparable throughput.

Goal: rebuild the entire two-layer kernel stack, in miniature, with real performance: newt, a mini-Triton (Triton was the original genus name for newts, so a newt is literally a small triton), and deuteron, a mini-Helion (a deuteron is a lighter atomic nucleus than a helion). Success means memory-bound kernels at parity with real Triton, matmul at a respectable, honestly-accounted fraction on tensor cores, and a codebase readable in an afternoon.

Part 3How newt works

3.1 The shortest path to machine code

A mini cannot carry MLIR and LLVM, and it does not need to: NVIDIA ships a C++ compiler as a library, NVRTC (NVIDIA Runtime Compilation). newt parses your Python function into an AST (abstract syntax tree, the parsed structure of the code), types every value with a shape, dtype and layout, emits a CUDA C++ source string, compiles it in-process with NVRTC into a cubin (compiled GPU binary), and launches it through the raw CUDA driver API using ctypes (Python's built-in foreign-function interface). No files, no external compiler, no wrapper packages.

3.2 The layout: where a block lives

That one problem: a Triton-style block like "1,024 floats" must be spread across the threads of a block. newt uses a single fixed rule, the group-cyclic layout: consecutive elements are dealt to threads in 16-byte groups, round-robin.

Triton proves contiguity with static compiler analysis. newt instead emits a tiny runtime check per group ("are these offsets consecutive, aligned, all unmasked?") and branches to the 128-bit fast path: a few integer compares against a ~500-cycle memory access. That single trick took vector-add from 82% of Triton to parity.

3.3 Reductions and broadcasting

nl.sum(x) combines values held by different threads in three stages: each thread reduces its own registers; each warp reduces its 32 lanes with a butterfly exchange (__shfl_xor_sync, lanes trading values directly with no memory involved, 5 steps for 32 lanes); one value per warp meets the others in shared memory. Broadcasting between different-sized blocks stages the smaller operand through a reusable shared-memory scratch arena with barriers around it; reshapes that do not move data are free.

3.4 The matmul story

Matrix multiply is compute-bound: the arithmetic outweighs the memory traffic, so the game flips from "save bandwidth" to "never let the tensor cores stall". Four mechanisms, added one commit at a time:

1. Tiles. Each thread block owns one BM x BN tile of the output and loops over K in BK steps, staging A and B tiles in shared memory. All three sizes are constexpr (compile-time constants), which is what makes them tunable.

2. Raw tensor-core PTX. fp16/bf16 dots compile to ldmatrix (a warp cooperatively loads 8x8 matrix fragments from shared memory into registers) and mma.sync.m16n8k16 (one warp multiplies a 16x16 by a 16x8 fragment in one instruction), written as inline PTX assembly. The accumulator's register layout is documented by NVIDIA, so newt can convert it to and from the normal layout whenever the kernel does elementwise math on it; that conversion is what makes the fused flash-attention example possible.

3. Swizzled shared memory. Shared memory has 32 banks; simultaneous hits on one bank serialize (bank conflicts). newt permutes each row's 16-byte chunks by XOR with the row index: free at runtime (it is just index math, applied identically by the writer and the reader) and it makes every ldmatrix access conflict-free, replacing wasteful padding.

4. The pipeline ring. The most important one, shown below.

Plus a final twist: within one k-step, fragment loads and math also form a dependency chain (inline assembly pins instruction order). newt double-buffers fragments so step k+1's ldmatrix issues before step k's math, hiding shared-memory latency behind the tensor cores. That last change took the cold-start matmul from 96 to 110 TFLOP/s (trillion floating-point operations per second).

Part 4How deuteron works

deuteron is ~700 lines doing to newt what Helion does to Triton. You write tiles; it writes kernels:

@dt.kernel
def matmul(x, y, out):
    for tile_m, tile_n in dt.tile([x.shape[0], y.shape[1]]):   # launch grid
        acc = dt.zeros([tile_m, tile_n], dtype=dt.float32)
        for tile_k in dt.tile(x.shape[1]):                     # k-loop
            acc += x[tile_m, tile_k] @ y[tile_k, tile_n]       # tensor cores
        out[tile_m, tile_n] = acc

Printing matmul.to_newt_source(...) shows the generated kernel, and it looks exactly like the hand-written tutorial matmul: the most satisfying demo in the repository.

Part 5Results

Same machine (RTX PRO 5000 Blackwell laptop GPU), same kernel source, same tuning sweep for newt and triton-windows; torch calls NVIDIA's hand-tuned cuBLAS/cuDNN libraries.

The remaining gap is Triton's finest-grained scheduling: strength-reducing address computations across loop iterations and specializing warps into producer/consumer roles. Known, documented, out of scope for a mini on purpose. tf32 (a 19-bit format used for fp32 matmuls) still uses NVIDIA's higher-level WMMA API and sits near 45%: mechanical future work.

Correctness got equal billing: 176 tests against PyTorch references and three adversarial review campaigns (hundreds of targeted GPU micro-programs plus a symbolic simulation of the pipeline state machine). Every confirmed finding was fixed with a regression test, and the git history doubles as the build log: each compiler stage is a self-contained commit.
